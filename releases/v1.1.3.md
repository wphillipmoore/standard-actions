
# Release 1.1.2 (2026-03-01)

## Bug fixes

- **fix version-bump regex and correct VERSION to 1.1.2 (#136)**
The capture group in version-regex was preserving the major.minor. prefix, then {version} inserted the full next version, producing 1.1.1.1.2 instead of 1.1.2. Removed the capture group so the entire match is replaced with the next version.

- **create placeholder releases/index.md when no releases exist (#148)**
Pre-release repositories have releases/index.md in their mkdocs nav but no release note files yet. With strict mode enabled, mkdocs aborts on the missing file. Generate a placeholder index when no releases/*.md files are found.

- **install Trivy binary directly to avoid setup-trivy cache fallback failure (#152)**
The aquasecurity/trivy-action setup-trivy mechanism has a broken cache miss fallback that attempts to clone the trivy repo from source, which fails with 'couldn't find remote ref refs/heads/main'. This replaces setup-trivy with a direct binary download from GitHub releases and passes skip-setup-trivy: true to trivy-action, giving us full control over the Trivy version.

- **use setup-trivy with cache disabled instead of manual binary download (#153)**
The previous fix attempted a manual curl download with an incorrect version/URL. This uses the official setup-trivy action with cache disabled to avoid the broken cache miss fallback, then passes skip-setup-trivy to trivy-action.

- **replace aquasecurity GitHub actions with Docker-based Trivy execution (#154)**
The aquasecurity/trivy GitHub repo is empty (HTTP 409: 'Git Repository is empty'), causing both setup-trivy and trivy-action to fail when attempting to clone it. This replaces all aquasecurity GitHub actions with direct docker run commands using the aquasec/trivy Docker Hub image, which is still available and up to date (v0.69.1). A new trivy-image input allows pinning the version.

- **add ci: type-check to required checks matrix (#143)**
Add ci: type-check as required for Python and Ruby repos. Type checking uses the ci: prefix because it is static analysis, not runtime test execution.

- **update tag protection ruleset docs for rolling tag support (#158)**

## Features

- **add rolling vX.Y minor tags to tag-and-release action (#149)**
When a new version tag is created (e.g. v1.2.3), a rolling minor tag (e.g. v1.2) is force-created and force-pushed, allowing consumers to pin @v1.2 and automatically receive patch releases.
